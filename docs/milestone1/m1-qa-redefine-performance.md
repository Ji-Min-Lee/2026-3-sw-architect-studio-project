## P-1: 실시간 오디오 처리 성능 (Real-Time Performance)

| 항목 / Item | 내용 / Detail |
|------------|--------------|
| **Source** | 기계식 시계 (음향 신호 발생) / Mechanical watch (acoustic signal source) |
| **Stimulus** | USB 센서를 통해 연속적인 음향 샘플 스트림 입력 / Continuous audio sample stream via USB sensor |
| **Artifact** | 신호 수집 → 처리 → 분석 파이프라인 (GUI 표시 지연은 QAS-3에서 관리) / Signal capture → processing → analysis pipeline (GUI display latency managed by QAS-3) |
| **Environment** | - 하드웨어: Raspberry Pi 5 (8GB RAM), Ubuntu 24.04 / Hardware: Raspberry Pi 5 (8GB RAM), Ubuntu 24.04<br>- 입력: USB PnP 마이크 연결, Live 모드 실행 중 / Input: USB PnP microphone connected, running in Live mode<br>- 측정 조건: 28,800 BPH 기계식 시계, 연속 10분 이상 / Measurement: 28,800 BPH mechanical watch, continuous ≥ 10 min<br>- 부하: Qt GUI 렌더링·오디오 처리·DSP 파이프라인 동시 실행 / Load: Qt GUI rendering, audio processing, and DSP pipeline running concurrently |
| **Response** | ALSA 드라이버가 생성하는 각 오디오 블록을 버퍼 오버플로 없이 수신하고, 다음 블록이 도착하기 전에 DSP 파이프라인(필터링 → T1/T3 이벤트 감지 → Rate·Amplitude·Beat Error 계산)을 완료한다 / Receive each audio block from the ALSA driver without buffer overflow and complete the DSP pipeline (filtering → T1/T3 event detection → Rate·Amplitude·Beat Error computation) before the next block arrives |
| **Response Measure** | - **Objective**: 96,000 sps 처리 유지 / sustain 96,000 sps<br>- **Minimum**: 48,000 sps (이 이하면 프로젝트 실패 / below this = project failure)<br>- **Stretch**: 192,000 sps<br>- Dropped audio block: **0**<br>- *(잠정값 — EX-01 결과로 확정 / Provisional — confirmed by EX-01)* |

> **한국어**
>
> TimeGrapher는 Raspberry Pi 5에서 USB 마이크를 통해 시계 음향 신호를 실시간으로 수집하고, ALSA 드라이버가 생성하는 연속 오디오 블록을 끊김 없이 처리해야 한다. SPS(Samples Per Second)는 단순한 샘플레이트 설정이 아니라 T1 감지 정밀도·초당 버퍼 처리 횟수·DSP 연산량을 동시에 결정하는 핵심 파라미터다. 목표는 96,000 sps를 유지하면서 Dropped Block = 0을 달성하는 것이며, RPi가 이를 감당하지 못하면 48,000 sps로 폴백하여 최소 동작을 보장한다. GUI 표시 지연은 QAS-3 Low Latency에서 별도 관리한다.
>
> **English**
>
> TimeGrapher must capture acoustic signals from a mechanical watch via USB microphone on Raspberry Pi 5 and process the continuous audio blocks generated by the ALSA driver without interruption. SPS (Samples Per Second) is not merely a sample rate setting — it is the single parameter that simultaneously determines T1 detection precision, buffer processing frequency per second, and DSP computation load. The goal is to sustain 96,000 sps with Dropped Block = 0; if the RPi cannot meet this target, the system falls back to 48,000 sps to guarantee minimum viable operation. GUI display latency is managed separately under QAS-3 Low Latency.

## SPS ↔ 성능 관계 / SPS ↔ Performance Relationship

**한국어**

SPS가 높을수록 T1 감지 정밀도는 올라가지만 ALSA 버퍼 처리 빈도와 DSP 연산량이 함께 증가한다. 아래 수치는 잠정값이며 EX-01 실측 전까지 확정 불가.

| SPS | 초당 버퍼 처리 횟수 / Buffers/sec | T1 감지 정밀도 / Detection resolution | 비고 / Note |
|-----|:--------------------------------:|:------------------------------------:|------------|
| 48,000 | ~50 | 20.8 µs / sample | 최소 동작 기준 (폴백) / Minimum viable (fallback) |
| **96,000** | **~100** | **10.4 µs / sample** | **목표 / Target** |
| 192,000 | ~200 | 5.2 µs / sample | RPi 성능 미확인 / Unverified on RPi |

**English**

Higher SPS improves T1 detection precision but proportionally increases ALSA buffer processing frequency and DSP computation load. Values below are provisional until EX-01 measurement.

*→ EX-01에서 SPS별 CPU%, Dropped Blocks를 측정 (end-to-end latency는 QAS-3 실험에서 별도 측정) / EX-01 measures CPU% and Dropped Blocks per SPS tier (end-to-end latency measured separately in QAS-3 experiment)*

## Risk

| 리스크 / Risk | 설명 / Description | 영향 / Impact |
|--------------|-------------------|--------------|
| **96k에서 Dropped Block 발생** | RPi 5가 초당 ~100 블록 처리를 감당하지 못할 경우 ALSA 버퍼 오버플로 발생 | T1/T3 이벤트 시간 정보 소실 → Rate·Amplitude 오차 증가 / Loss of T1/T3 timing information → increased Rate·Amplitude error |
| **GUI 렌더링과 오디오 처리 경합** | 비가시 탭까지 매 프레임 replot() 호출 시 UI 스레드가 오디오 처리를 지연 | Foreground SPF 급증 → Dropped Block 간접 유발 / Foreground SPF spike → indirect cause of Dropped Blocks |
| **DSP 연산 집중** | 96k sps에서 FFT·RLS·Rolling Average 연산이 한 스레드에 집중 | 단일 코어 병목 발생 시 버퍼 처리 지연 → Dropped Block / Single-core bottleneck delays buffer processing → Dropped Block |
| **48k 폴백 후 정밀도 저하** | SPS를 48k로 낮추면 T1 감지 정밀도가 10.4µs → 20.8µs로 절반 감소 | Beat Error 측정 분해능 저하, QAS-2 목표(< 0.1ms) 충족 여부 재확인 필요 / Reduced Beat Error resolution; QAS-2 target (< 0.1ms) needs re-verification |
| **스케줄러 지터** | Linux 범용 스케줄러에서 오디오 스레드 우선순위가 낮아 간헐적 처리 지연 발생 | 산발적 Dropped Block → 장기 Rate 측정 오차 누적 / Sporadic Dropped Blocks → accumulated long-term Rate measurement error |

## Tactic / Pattern

| 전술 · 패턴 / Tactic · Pattern | 적용 이유 / Rationale |
|-------------------------------|----------------------|
| **Pipe-and-Filter** | Acquisition → Processing → Detection → Calculation → Visualization 5단계 분리. 각 단계의 처리 부하를 독립적으로 측정·교체 가능 / 5-stage separation enables independent measurement and replacement of each stage's processing load |
| **Lock-Free Ring Buffer** | 오디오 스레드(생산자)와 처리 스레드(소비자) 사이에 락 없는 큐 배치 → 뮤텍스 경합 제거, Dropped Block 방지 / Lock-free queue between audio thread (producer) and processing thread (consumer) eliminates mutex contention, preventing Dropped Blocks |
| **Graceful Degradation** | 96k sps 유지 불가 시 자동으로 48k sps로 전환. 측정 정밀도는 낮아지지만 Dropped Block 없는 연속 동작 보장 / Auto-switch to 48k sps when 96k is unsustainable; lower precision but zero Dropped Block guaranteed |
| **Lazy Rendering (isVisible())** | 현재 선택된 탭의 그래프만 replot() 호출. 비가시 탭은 데이터 누적만 수행 → UI 스레드 부하 최소화 / Only the active tab calls replot(); invisible tabs accumulate data only → minimizes UI thread load |
| **Priority Scheduling** | 오디오 처리 스레드에 높은 스케줄링 우선순위 부여 → Linux 범용 스케줄러 지터로 인한 Dropped Block 예방 / Elevating audio thread scheduling priority prevents Dropped Blocks caused by Linux general-purpose scheduler jitter |
