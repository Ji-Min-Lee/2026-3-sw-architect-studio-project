# Architectural Drivers — TimeGrapher

**Milestone**: M1 | **Due**: 2026-06-09 | **Updated**: 2026-06-03

---

## 1. Project Context & Objectives

TimeGrapher is a system that analyzes acoustic signals (beat noise) generated by the escapement vibration of a mechanical watch to diagnose watch performance in real time. It captures the tick-tock sound from a microphone, detects A(T1)·C(T3) events, computes Rate, Amplitude, and Beat Error, and displays them in a GUI.

| Goal | Description |
|------|-------------|
| Real-Time Diagnosis | Collect, process, and display acoustic signals in real time on Raspberry Pi 5 |
| Accurate Measurement | Rate, Amplitude, Beat Error values match the WeiShi No.1000 reference device |
| Extensible Architecture | New graphs and analysis features can be added without modifying existing code |
| Architecture Principles | System design applying CMU MSE software architecture principles |

## 3. Quality Attribute Scenarios

### 3.1 Priority & Rationale

The five QAs are prioritized in order of the greatest constraint they impose on architectural decisions.

| Rank | QA | One-Line Rationale |
|:----:|----|-------------------|
| **1** | Real-Time Performance | Prerequisite for all other QAs — without it, there is no data |
| **2** | Measurement Accuracy | The reason this project exists; most directly shapes architectural decisions (sps, detection algorithm) |
| **3** | Low Latency | Hard threshold defined by beat period; exceeding it causes functional failure; verifiable only through experiment |
| **4** | Correctness | Structurally guaranteed by Accuracy + single data source; not an independent architectural goal |
| **5** | Extensibility | Developer convenience, but directly controls schedule risk for implementing 11 graphs |

### QAS-3: Low Latency ★ Priority 3

| Item | Detail |
|------|--------|
| **Source** | Mechanical watch (beat event source) |
| **Stimulus** | Audio sample block capture starts from microphone |
| **Artifact** | Full pipeline: capture → beat detection → computation → GUI rendering |
| **Environment** | Raspberry Pi 5, real-time operating state |
| **Response** | Waveform, markers, and computed values for captured samples displayed in GUI |
| **Response Measure** | See latency target derivation below |

#### Latency Target Derivation

**Core constraint**: If end-to-end latency exceeds the beat period, the previous beat display cannot complete before the next beat is processed, causing functional failure.  
**80% safety margin** applied to absorb OS jitter and Qt rendering variation.

| BPH | Beat Period | 80% margin → end-to-end target | ① capture→process (70%) | ② process→display (30%) |
|-----|------------|--------------------------------|------------------------|------------------------|
| 28,800 | 125 ms | 100 ms | 70 ms | 30 ms |
| 36,000 | 100 ms | 80 ms | 56 ms | 24 ms |
| **43,200** | **83 ms** | **66 ms** | **46 ms** | **20 ms** |

> | Option | Range | end-to-end | ① | ② | Note |
> |:------:|-------|-----------|---|---|------|
> | **Option A** *(current target)* | ~28,800 BPH | **< 100 ms** | **< 70 ms** | **< 30 ms** | Baseline target |
> | **Option B** *(stretch)* | ~43,200 BPH (worst-case) | < 66 ms | < 46 ms | < 20 ms | Consider after EX-01 |
>
> *Current target is Option A (28,800 BPH baseline). Option B (43,200 BPH stretch) will be considered for upgrade after EX-01 results.*

> ⚠️ **Team decision required — supported BPH range not yet confirmed (to be decided after verifying available watches)**

**Common Constraints**: Dropped audio block: 0, Missed beat: 0  
*(to be confirmed after [EX-01](./planned-experiments.md#ex-01-rpi-performance-benchmark))*

### QAS-5: Extensibility ★ Priority 5

| Item | Detail |
|------|--------|
| **Source** | Developer (adding new graph or analysis feature) |
| **Stimulus** | Request to add new graph, filter, or display mode to existing codebase |
| **Artifact** | Full GUI module structure (capture / processing / computation / display layers) |
| **Environment** | Development environment (PC or RPi), within project timeline |
| **Response** | New features can be added and tested independently without major redesign of existing modules |
| **Response Measure** | Files changed to add 1 new graph: **≤ 3** (see derivation below)<br>Display layer can be added without modifying existing computation logic |

#### "≤ 3 Files" Derivation

**≤ 3 is the theoretical minimum in an ideal plugin-style architecture.**

| Change Type | Target | Files |
|-------------|--------|:-----:|
| ① Create new graph widget file | `GraphFoo.cpp` / `GraphFoo.h` | 1 |
| ② Register new tab in panel | `TabPanel.cpp` (existing modification) | 1 |
| ③ Wire data subscription | `DataBroker.cpp` (existing modification) | 1 |
| **Total** | | **3** |

- **Floor**: Cannot add a graph with 0 file changes — at least 1 new file must be created
- **If exceeded**: ≥ 4 files signals insufficient modularization (e.g., God Object forces modification)
- **Schedule link**: With 11 graphs due in Weeks 3–4, this constraint numerically enforces Extensibility and directly controls schedule risk
- **Demo verification**: "1 graph added = 3 files changed" is a concrete, measurable criterion for demonstrating Extensibility at the final demo
