# 리스크 평가 — TimeGrapher

---

이 문서는 세 가지 질문에 순서대로 답합니다:

1. 기술적·비기술적 리스크는 무엇이며, 확률과 영향은 어떻게 평가되는가?
2. 미결 질문/이슈가 프로젝트 결과에 영향을 줄 사항과 명확하게 연결되어 있는가?
3. 미결 질문/이슈를 해결하기 위한 실행 방안이 식별되었는가?

---

## 1. 리스크 평가 기준

각 리스크는 확률과 영향을 H / M / L 3단계로 평가합니다.

| 수준 | 확률 기준 | 영향 기준 |
|:---:|---------|---------|
| **H (높음)** | 실험 없이는 보장 불가 / 사전 지식으로 이미 발생이 예상됨 | QA 목표 달성 불가 → 팀 주요 목표(정확한 측정) 또는 납품을 직접 위협 |
| **M (중간)** | 설계 결정으로 완화 가능하나 실험 결과에 따라 악화될 수 있음 | 일부 QA 목표 저하 / 개발 속도 감소 |
| **L (낮음)** | 알려진 완화책이 있거나 확률이 낮음 | 부분적 기능 부족 또는 특정 환경에만 영향 |

**전체 리스크 = max(확률, 영향)**. 두 축 중 하나라도 H이면 리스크는 전체적으로 H로 평가됩니다.

### TR-01 — RPi 5가 96k sps에서 블록 드롭 = 0을 달성할 수 없음

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | DSP 파이프라인이 96k sps 블록 주기(~10ms)를 초과하면 링 버퍼가 오버플로하여 블록 드롭 발생 |
| **확률: H** | RPi 5가 Qt GUI + DSP를 동시에 실행하면서 96k sps를 유지할 수 있는지 실험 전에는 미검증 |
| **영향: H** | 블록 드롭 → T1/T3 타임스탬프 손실 → 레이트·진폭·박자 오차 계산 불가 → **팀 주요 목표 붕괴**. 모든 QA의 전제 조건 |
| **완화 전술** | ① 점진적 성능 저하: 96k 달성 불가 시 48k sps로 자동 폴백 (블록 드롭 = 0 보장, 낮은 해상도) ② 잠금-없는 링 버퍼: 뮤텍스 제거로 DSP 지연 방지 |
| **잔여 위험** | 48k sps 폴백 시 T1 검출 해상도가 20.8 µs/샘플로 저하 — 36,000 / 43,200 BPH 지원이 불가능해짐 |
| **연계 사항** | OI-P1 → **EXP-01**로 해결 |

---

### TR-02 — Linux 스케줄러 지터로 인한 간헐적 블록 드롭

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | Linux 범용 스케줄러가 오디오 스레드 우선순위를 낮게 처리하여 간헐적 블록 주기 위반 → 산발적 블록 드롭 → 누적된 장기 레이트 측정 오류 |
| **확률: M** | Linux 범용 스케줄러의 알려진 특성; 오디오 스레드 우선순위를 기본값으로 두면 발생 |
| **영향: H** | 산발적 블록 손실로 시간이 지남에 따라 레이트 측정 오류 누적 → 정확도 저하; 재현이 어려워 디버깅 비용 증가 |
| **완화 전술** | 우선순위 스케줄링: 오디오 스레드 우선순위 상승 (`SCHED_RR` 또는 `SCHED_FIFO`)으로 Linux 스케줄러 지터 흡수 |
| **잔여 위험** | RPi에서의 효과는 EXP-01을 통해 공동 검증해야 함 |
| **연계 사항** | EXP-01 결과의 산발적 블록 드롭 패턴에서 간접적으로 확인 |

---

### TR-03 — 종단간 지연이 박자 주기를 초과함 (RPi + Qt 미검증)

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | 전체 파이프라인(오디오 캡처 → DSP → Qt GUI 렌더링)이 100ms(28,800 BPH에서 125ms 박자 주기의 80%) 이내에 완료되지 못하면 실시간 표시 실패 |
| **확률: H** | RPi 5 + Qt6에서 ① 캡처→처리(< 70ms) + ② 처리→표시(< 30ms) 모두 충족 가능한지 EXP-02 전에는 미검증; QAudioSource 라이브 콜백 주기 자체도 미확정 |
| **영향: H** | 지연 위반 → 이전 박자 표시 완료 전에 다음 박자 처리 시작 → 실시간 표시 붕괴 → T1/T3 타임스탬프 정확도에 간접 영향 |
| **완화 전술** | ① 동시성 도입 (오디오/DSP/GUI 스레드 분리) ② 잠금-없는 링 버퍼 ③ 지연 렌더링 (활성 탭만 렌더링) ④ 계산 오버헤드 감소 |
| **연계 사항** | OI-L1 (콜백 주기 미확인), OI-L2 (11-탭 렌더링 오버헤드) → **EXP-02**로 해결 |

---

### TR-04 — 11-탭 동시 렌더링으로 ② 처리→표시 > 30ms

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | 11개 그래프 탭이 완성되면 Qt 메인 스레드 렌더링 부하로 인해 ② 처리→표시가 30ms를 초과할 수 있음 |
| **확률: M** | 1-2개 탭은 괜찮을 수 있지만; 11개 탭 이후 누적 부하는 가능성이 있으나 미검증 |
| **영향: M** | 종단간 위반을 초래하면 TR-03과 동일한 결과; 그러나 지연 렌더링으로 완화 가능 |
| **완화 전술** | 지연 렌더링: 활성 탭에서만 `paintEvent()` 실행; 비활성 탭은 데이터 업데이트만 하고 렌더링 지연 |
| **연계 사항** | OI-L2 → EXP-02에서 1-탭 vs. 11-탭 구성 비교로 확인 |

---

### TR-05 — 검출기 기본 파라미터가 주변 노이즈 하에서 실패

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | `Detector.cpp`의 기본 `onset_fraction` (0.03) / `min_peak_fraction` (0.20)이 주변 노이즈 하에서 박자 검출 품질을 저하시킬 수 있음 |
| **확률: M** | 적응형 임계값 알고리즘이 구현되어 있으나 기본 파라미터가 3가지 노이즈 조건(낮음/중간/높음) 하에서 유효한지 실험적으로 미검증 |
| **영향: M** | Δ 레이트 / Δ 진폭 / Δ 박자 오차 증가 → 팀 주요 목표의 부분 저하; 신뢰성은 감소하지만 측정이 완전히 실패하지는 않음 |
| **완화 전술** | 적응형 임계값 (noise_floor = 최근 256ms의 75번째 백분위수, reference_peak = 최근 16개 박자의 중앙값) + EXP-03으로 최적 파라미터 확정 |
| **연계 사항** | OI-C1 → **EXP-03** (3 노이즈 조건 × 검출기 파라미터 조합)으로 해결 |

### TR-06 — 옵저버 리팩토링 후 잔존 결합으로 ≤ 3-파일 제약 위반

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | 옵저버 패턴 리팩토링 후 표현 계층과 신호 처리 / 수집 계층 간의 잔존 결합이 지속되어, 새 그래프 추가 시 3개 이상의 파일을 변경하게 됨 |
| **확률: M** | 완전한 계층 분리를 위해서는 기존 코드에 대한 깊은 이해가 필요; 리팩토링 중 부분적 결합이 남을 가능성이 높음 |
| **영향: M** | ≤ 3-파일 제약 위반 → 이후 개발 주에서 그래프 추가 비용 증가, 병렬 개발 일정 위험 현실화 |
| **완화 전술** | 의존성 제한 (계층화 아키텍처): 표현 계층이 도메인 계층(MeasurementEngine 인터페이스)만 참조 가능하도록 규칙 강제 |
| **연계 사항** | 옵저버 패턴 리팩토링 완료 후 ≤ 3-파일 실험으로 직접 검증 |

### TR-07 — God Object 분해 중 기존 그래프의 회귀

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | God Object에서 계층화 시그널-슬롯 구독 모델로의 구조적 전환 중 기존 그래프(트레이스, 박자 오차 등)의 부작용 발생 |
| **확률: M** | 대규모 구조적 리팩토링의 일반적인 위험; 기존 코드베이스(`TimeGrapher_v10.5`)에 자동화된 회귀 테스트가 없어 감지가 어려움 |
| **영향: H** | 회귀 → 롤백 → God Object 구조로 복귀 → QAS-5 (확장성) 포기 → 11개 그래프 병렬 개발 완전 붕괴 |
| **완화 전술** | ① 점진적 리팩토링 (기능별로 분해, 각 단계에서 검증) ② 리팩토링 전후 동일한 시계와 동일한 조건에서 레이트·진폭·박자 오차 값을 수동으로 비교 ③ 핵심 계산 로직(레이트·진폭·박자 오차)에 대한 단위 테스트 작성 — 리팩토링 전 기준선 수립, 리팩토링 후 동일 테스트 실행으로 회귀 자동 감지 |
| **연계 사항** | 별도 미결 사항 없음 — 위험이 리팩토링 실행 단계에 내재되어 있음 |

### TR-08 — 새 그래프에 MeasurementEngine이 현재 제공하지 않는 데이터 필요

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | 새 그래프(예: 스펙트로그램)가 MeasurementEngine이 현재 발행하지 않는 원시 오디오 데이터를 필요로 하여 수정 범위가 전처리 계층까지 확장됨 |
| **확률: L** | 11개 그래프 대부분은 레이트·진폭·박자 오차 파생값만 필요로 함; 원시 오디오 의존 그래프를 범위에서 제외하면 확률이 낮음 |
| **영향: M** | ≤ 3-파일 제약 위반 + 전처리 계층 수정으로 인해 TR-07 회귀 위험 동시 발생 |
| **완화 전술** | 11개 그래프 목록을 사전에 검토하여 MeasurementEngine의 현재 출력이 모든 데이터 필요를 충족하는지 확인; 충족하지 않으면 먼저 도메인 계층 인터페이스 확장 |
| **연계 사항** | 그래프 구현 계획이 확정될 때 사전 구현 검토를 통해 해결 |

### TR-09 — 신호 품질 경고 임계값이 실제 환경과 불일치

| 항목 | 세부 사항 |
|-----|---------|
| **설명** | `⚠ 신호 없음` / `⚠ 잡음 있는 신호`의 경고 발생/해제 임계값(N·M초, 노이즈/신호 비율)이 실제 환경의 주변 노이즈 수준과 맞지 않아 오탐 또는 누락 경고 발생 가능 |
| **확률: M** | 임계값이 환경에 의존적; 실험실과 현장 환경 간에 노이즈 플로어가 크게 다를 수 있음 |
| **영향: L** | 사용자 경험 저하 — 오탐은 측정에 대한 신뢰를 약화시키거나 경고를 무시하는 습관을 만들 수 있음; 측정 정확도에는 직접적인 영향 없음 (QAS-3 정확성과 독립적) |
| **완화 전술** | 하트비트 패턴 (기존 A/C 이벤트 재사용): 추가 검출 로직 없이 N초 동안 박자 이벤트가 없으면 `⚠ 신호 없음` 표시 — 임계값 조정이 단일 파라미터로 단순화됨 |
| **연계 사항** | OI-U1 (N·M 값), OI-U2 (노이즈/신호 임계값) → 시계 제거/복원 실험 + 다중 환경 임계값 탐색으로 해결 |

### NTR-01 — 5주 내 11개 그래프 병렬 구현에 대한 일정 과부하
