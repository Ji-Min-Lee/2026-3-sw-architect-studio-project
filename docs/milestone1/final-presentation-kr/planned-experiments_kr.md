## 실험 EXP-01: RPi 실시간 성능 — 블록 드롭 측정

### 결과 및 권고사항

(미완료 — 실험 수행 후 기록 예정.)

---

### 목표

Raspberry Pi 5에서 Qt GUI + DSP 파이프라인을 48,000 / 96,000 / 192,000 sps로 동시에 실행했을 때 블록 드롭(링 버퍼 오버플로)이 발생하지 않는지 검증합니다.

**기술적 질문**: "RPi 5가 Qt GUI + DSP를 동시에 실행하면서 96,000 sps에서 블록 드롭 = 0을 달성할 수 있는가? 그렇지 않다면 안정적으로 처리할 수 있는 최대 sps는 얼마인가?"

이 실험의 결과는 다음 사항에 직접 영향을 줍니다:
- QAS-1 응답 측도 확정 (현재 목표: 96,000 sps는 잠정적)
- 점진적 성능 저하 폴백 임계값 확정 (48k sps 폴백 트리거 여부)
- EXP-02와 EXP-03의 전제 조건 — SPS 방향이 결정되기 전까지 지연 측정 및 파라미터 조정은 의미 없음

---

### 상태

**계획됨**

---

### 예상 결과물

- 각 조건별 블록 드롭 횟수 표 (48k / 96k / 192k sps, 10분 연속 실행)
- 달성 가능한 최대 sps 결론 및 확정된 QAS-1 응답 측도
- 점진적 성능 저하 폴백 결정 (96k 달성 가능 여부 → 48k 폴백 필요 여부)
- 우선순위 스케줄링(`SCHED_RR` / `SCHED_FIFO`) 적용 전후 비교

---

### 필요 자원

| 자원 | 세부 사항 |
|-----|---------|
| 하드웨어 | Raspberry Pi 5 (8GB RAM), USB 오디오 센서 (마이크로폰), 8" 터치스크린 |
| 소프트웨어 | Ubuntu 24.04, Qt6, ALSA, `TimeGrapher_v10.5` 코드베이스 |
| 기계식 시계 | 28,800 BPH 시계 1개 (라이브 캡처 오디오 소스) |
| 측정 도구 | 링 버퍼 오버플로 카운터 주입, `chrono` 타임스탬프, Linux `perf` |
| 노력 | ~1–2 인일 (설정 + 측정 + 분석) |

---

### 실험 절차

1. **환경 설정**: RPi 5에서 Ubuntu 24.04 + Qt6 빌드 환경 구성. USB 마이크로폰 앞에 28,800 BPH 시계 배치.
2. **링 버퍼 오버플로 카운터 주입**: `AudioCapture` 모듈에 `dropped_block_count` 변수 추가; 오버플로 이벤트 시 증가.
3. **sps 조건 설정**: `QAudioFormat::setSampleRate()`를 사용하여 48,000 / 96,000 / 192,000 sps를 순서대로 설정.
4. **10분 연속 실행**: 각 sps 조건에서 Qt GUI + DSP(필터링 + T1/T3 감지 + 레이트·진폭·박자 오차 계산)를 10분간 동시 실행.
5. **우선순위 스케줄링 비교**: `SCHED_RR` 적용 전후로 동일 조건을 반복하여 Linux 스케줄러 지터 효과 측정 (TR-02 검증).
6. **결과 기록**: sps별 블록 드롭 횟수 표 작성. 96k 달성 가능 여부 확인 → 확정된 QAS-1 응답 측도 기록.
7. **폴백 결정**: 96k 달성 불가 시, 점진적 성능 저하 폴백(48k sps 자동 전환) 결정을 아키텍처 위원회에 보고.

- **전제 조건**: RPi 5 하드웨어 설정 + 기계식 시계 연결
- **완료 전까지 차단되는 작업**: EXP-02(지연 측정), EXP-03(파라미터 조정) — SPS 방향이 결정되기 전까지 두 실험의 측정 조건 미정

## 실험 EXP-02: 종단간 지연 — 3-구간 타임스탬프 측정

### 결과 및 권고사항

(미완료 — 실험 수행 후 기록 예정.)

---

### 목표

전체 TimeGrapher 파이프라인(오디오 캡처 → DSP → Qt GUI 렌더링)의 종단간 지연을 3개 구간으로 나누어 측정합니다. 각 구간의 병목을 파악하고 28,800 BPH에서 종단간 < 100ms가 달성 가능한지 검증합니다.

**이 실험이 답해야 하는 기술적 질문**:
- "실제 QAudioSource 라이브 캡처 콜백 주기는 얼마인가? (OI-L1)"
- "11개 탭이 동시에 렌더링할 때 ② 처리→표시가 30ms를 초과하는가? (OI-L2)"
- "28,800 BPH 달성 후 목표를 36,000 / 43,200 BPH로 높일 수 있는가? (OI-L3 — 데이터 수집만; 최종 해결은 EXP-05로 이연)"

이 실험의 결과는 다음 사항에 직접 영향을 줍니다:
- QAS-2 응답 측도 확정 (BPH 기반 지연 목표)
- 지연 렌더링 전술이 필수인지 결정 (OI-L2 결과 기반)
- BPH 확장 가능성에 대한 예비 데이터 (OI-L3 — EXP-05에서 최종 해결)

---

### 상태

**계획됨**

---

### 예상 결과물

- 지연 측정 표: 3 구간 × 3 sps 계층 × 2 탭 구성 (1-탭 / 11-탭), 평균 + 최악의 경우
- QAudioSource 콜백 주기 측정값 (OI-L1 해결)
- 지연 렌더링 필요성 결정 (OI-L2: 11탭에서 ② 구간이 30ms를 초과하는지 여부)
- BPH 확장 가능성에 대한 팀 보고 수치 (OI-L3)
- 확정된 QAS-2 응답 측도

---

### 필요 자원

| 자원 | 세부 사항 |
|-----|---------|
| 하드웨어 | Raspberry Pi 5 (8GB RAM), USB 오디오 센서, 8" 터치스크린 |
| 소프트웨어 | Ubuntu 24.04, Qt6, `TimeGrapher_v10.5` 코드베이스 |
| 기계식 시계 | 28,800 BPH 시계 (주요 목표); 추가 BPH 시계 (OI-L3 탐색 시) |
| 측정 도구 | 3개 지점(TS1/TS2/TS3)에서 `std::chrono::high_resolution_clock` 타임스탬프, Qt `QElapsedTimer` |
| 탭 구성 | 1-탭 빌드 vs. 11-탭 전체 빌드 — 두 바이너리 또는 런타임 탭 수 제어 |
| 전제 조건 | EXP-01 완료 (SPS 방향 결정), 옵저버 패턴 리팩토링 완료 |
| 노력 | ~2–3 인일 |

---

### 실험 절차

**타임스탬프 주입 지점 (3개 지점)**:

| 지점 | 위치 | 측정 구간 |
|-----|-----|---------|
| **TS1** | ALSA 콜백 수신 직후 (`audioDataAvailable()` 진입) | ① 시작 |
| **TS2** | T1/T3 이벤트 타임스탬프 확정 직후 (DSP 처리 완료) | ① 종료 / ② 시작 |
| **TS3** | Qt `paintEvent()` 완료 직후 (GUI 렌더링 완료) | ② 종료 = ③ 종료 |

**실험 절차**:

1. **타임스탬프 코드 주입**: TS1·TS2·TS3 위치에 `std::chrono::high_resolution_clock::now()` 삽입. 박자 이벤트마다 세 값을 로그 파일에 기록.
2. **1-탭 구성 측정**: 하나의 탭만 활성화된 빌드로; 각 sps 계층(48k/96k/192k) × 28,800 BPH 시계에서 5분간 실행. 구간별 평균 및 최악의 경우 수집.
3. **11-탭 구성 측정**: 11개 탭 모두 활성화된 동일 조건에서 반복. ② 구간(처리→표시)을 1-탭 기준선과 비교.
4. **콜백 주기 측정 (OI-L1)**: TS1 간격의 히스토그램을 그려 실제 QAudioSource 콜백 주기와 지터 분포 특성화.
5. **지연 렌더링 결정 (OI-L2)**: 11개 탭에서 ② 구간 > 30ms의 빈도 계산. 임계값 초과 시 지연 렌더링 전술 적용 결정 트리거.
6. **BPH 확장 검토 (OI-L3)**: 28,800 BPH 종단간 < 100ms 달성 시 36,000 / 43,200 BPH 시계로 동일 측정 반복하고 팀에 가능성 보고.
7. **결과 확정**: QAS-2 BPH 기반 지연 목표 표를 측정값으로 채우고; 응답 측도 확정.

---

### 기간

- **목표 완료**: 스프린트 2–3 이내 (참고: ~2026-06-12)
- **전제 조건**: ① EXP-01 완료 (SPS 방향 확정) ② 옵저버 패턴 리팩토링 완료 (탭 비교를 위한 11-탭 빌드 필요)
- **완료 기준**: 3-구간 × 2-탭 구성 측정 표 완성 + OI-L1·OI-L2 결론 기록 + QAS-2 응답 측도 확정

---

## 실험 EXP-03: 노이즈 조건 하에서 검출기 파라미터 최적화

### 결과 및 권고사항

(미완료 — 실험 수행 후 기록 예정.)

---

### 목표

세 가지 노이즈 조건(낮음 / 중간 / 높은 주변 노이즈)에서 `Detector.cpp`의 `onset_fraction`과 `min_peak_fraction` 파라미터 조합을 체계적으로 탐색하여 Δ 레이트 / Δ 진폭 / Δ 박자 오차를 최소화하는 최적 조합을 찾습니다.

**기술적 질문**: "낮음 / 중간 / 높은 노이즈 환경에서 레이트 / 진폭 / 박자 오차의 측정 편차(Δ)를 최소화하는 `onset_fraction`과 `min_peak_fraction`의 조합은 무엇인가? (OI-C1)"

이 실험의 결과는 다음 사항에 직접 영향을 줍니다:
- QAS-3 QA-C2 응답 측도 확정 (현재 잠정적, "EXP-03 이후 확정"으로 표시)
- `Detector.cpp`의 기본 파라미터 업데이트 (TR-05 해결)
- 실제 노이즈 조건 하에서 적응형 임계값 알고리즘 효과성 검증

---

### 상태

**계획됨**

---

### 예상 결과물

- 그리드 탐색 결과 표: `onset_fraction` × `min_peak_fraction` 조합 × 3 노이즈 조건, 셀당 Δ 레이트 / Δ 진폭 / Δ 박자 오차
- 노이즈 조건별 최적 파라미터 조합 + 권장 기본값
- 각 노이즈 조건 하에서 적응형 임계값 유효성 평가 (noise_floor = 최근 256ms의 75번째 백분위수, reference_peak = 최근 16개 박자의 중앙값)
- 확정된 QAS-3 QA-C2 응답 측도 (허용 Δ 임계값 포함)

---

### 필요 자원

| 자원 | 세부 사항 |
|-----|---------|
| 하드웨어 | Raspberry Pi 5, USB 오디오 센서, 8" 터치스크린 |
| 소프트웨어 | Ubuntu 24.04, Qt6, `TimeGrapher_v10.5` 코드베이스 |
| 기계식 시계 | 28,800 BPH 시계 1개 (파라미터 조정용 기준 시계) |
| 노이즈 환경 | 세 가지 주변 노이즈 수준 (예: 조용한 실험실 / 사무실 / 노이즈 추가) |
| 측정 도구 | 조건별 N회 반복에서 레이트·진폭·박자 오차 로그; Witschi 또는 동등한 기기의 기준값 |
| 전제 조건 | EXP-01 완료 (측정에 사용할 SPS 확정) |
| 노력 | ~2–3 인일 |

---

### 실험 절차

**파라미터 탐색 범위 (그리드 탐색)**:

| 파라미터 | 현재 기본값 | 탐색 범위 | 단계 |
|---------|:--------:|---------|:---:|
| `onset_fraction` | 0.03 | 0.01 ~ 0.10 | 0.01 |
| `min_peak_fraction` | 0.20 | 0.10 ~ 0.40 | 0.05 |

**노이즈 조건 정의**:

| 조건 | 환경 설명 | 예상 노이즈 플로어 |
|-----|---------|:--------------:|
| 낮은 노이즈 | 조용한 밀폐 실험실, 주변 장비 끔 | ~30 dB SPL |
| 중간 노이즈 | 일반 사무실, 공조 및 대화 노이즈 | ~50 dB SPL |
| 높은 노이즈 | 노이즈 소스 추가 (예: 음악 재생) | ~65 dB SPL |

**실험 절차**:

1. **기준선 수집**: 기본 파라미터(`onset_fraction=0.03`, `min_peak_fraction=0.20`)로 낮은 노이즈 환경에서 레이트·진폭·박자 오차를 30회 측정하여 기준선 수립.
2. **노이즈 조건 순서 고정**: 낮음 → 중간 → 높음. 각 노이즈 조건 변경 후 2분 안정화.
3. **그리드 탐색**: 각 노이즈 조건에서 모든 `onset_fraction` × `min_peak_fraction` 조합을 순서대로 적용. 조합별 레이트·진폭·박자 오차를 10회 측정.
4. **Δ 계산**: 각 조합에 대해 기준선(낮은 노이즈 + 기본 파라미터) 대비 Δ 레이트 / Δ 진폭 / Δ 박자 오차 계산.
5. **최적 조합 선택**: 세 가지 노이즈 조건에서 Δ 합계가 최소인 파라미터 조합 선택.
6. **적응형 임계값 검증**: 적응형 임계값 활성화 상태에서 최적 파라미터 조합이 갑작스러운 노이즈 전환(낮음→높음 전환)에서도 유효한지 확인.
7. **결과 기록**: 최적 파라미터 조합 + QAS-3 QA-C2 허용 Δ 값을 아키텍처 드라이버에 반영.

---

### 기간

- **목표 완료**: 스프린트 3–4 이내 (참고: ~2026-06-18)
- **전제 조건**: EXP-01 완료 (측정에 사용할 SPS 확정)
- **완료 기준**: 3-조건 × 전체 파라미터 조합 측정 표 완성 + 최적 파라미터 확정 + QAS-3 QA-C2 응답 측도 기록

## 실험 EXP-04: 신호 품질 경고 임계값 탐색

### 결과 및 권고사항

(미완료 — 실험 수행 후 기록 예정.)

---

### 목표

실제 환경에서 `⚠ 신호 없음` / `⚠ 잡음 있는 신호` 경고의 최적 발생/해제 임계값(N초, M초, 노이즈/신호 비율)을 측정하여 오탐과 누락 경고를 최소화합니다.

**기술적 질문**:
- "마이크로폰에서 시계를 제거한 후 몇 초 안에 `⚠ 신호 없음`이 나타나야 사용자가 즉시 알아챌 수 있는가? (OI-U1 — N 값)"
- "시계를 복원한 후 안정적인 신호 복구가 확인될 때까지 경고가 몇 초 안에 해제되어야 하는가? (OI-U1 — M 값)"
- "실제 환경에서 오탐 없이 `⚠ 잡음 있는 신호`를 트리거하는 노이즈/신호 비율 임계값은 무엇인가? (OI-U2)"

이 실험의 결과는 다음 사항에 직접 영향을 줍니다:
- QAS-4 응답 측도 확정 (N·M 값 현재 미해결)
- 하트비트 패턴 파라미터 설정 (N초 타임아웃 값)
- `⚠ 잡음 있는 신호` 임계값을 코드 상수로 확정

---

### 상태

**계획됨**

---

### 예상 결과물

- 시계 제거부터 `⚠ 신호 없음` 경고 표시까지의 시간 분포 (측정된 N초)
- 시계 복원부터 경고 해제까지의 시간 분포 (측정된 M초)
- 여러 노이즈 환경(낮음/중간/높음)에서 `⚠ 잡음 있는 신호` 임계값 후보 및 오탐률
- 확정된 QAS-4 응답 측도 (N·M 값 + 노이즈/신호 임계값)
- 권장 하트비트 파라미터 값

---

### 필요 자원

| 자원 | 세부 사항 |
|-----|---------|
| 하드웨어 | Raspberry Pi 5, USB 오디오 센서, 8" 터치스크린 |

| 기계식 시계 | 28,800 BPH 시계 1개 (제거/복원 작업 대상) |
| 노이즈 환경 | EXP-03과 동일 — 세 가지 주변 노이즈 수준 (낮음/중간/높음) |
| 측정 도구 | 경고 표시 타임스탬프 로그; 탐색을 위한 구성 가능한 하트비트 N초 파라미터 |
| 전제 조건 | 옵저버 패턴 리팩토링 완료 + `⚠ 신호 없음` / `⚠ 잡음 있는 신호` 경고 UI 구현 |

### 실험 절차

**파트 A — `⚠ 신호 없음` N·M 값 탐색 (OI-U1)**:

1. 하트비트 N 파라미터를 1 / 2 / 3 / 5초로 설정한 빌드를 준비합니다.
2. 각 N 설정에서 28,800 BPH 시계를 마이크로폰에서 제거하고 경고가 나타나기까지의 시간을 측정합니다 (10회 반복).
3. 시계를 복원하고 경고가 해제되기까지의 시간 M을 측정합니다 (10회 반복).
4. "너무 느림(사용자 인지 지연)"과 "너무 빠름(신호 변동으로 인한 오탐)" 균형을 고려하여 최적 N·M을 결정합니다.

**파트 B — `⚠ 잡음 있는 신호` 임계값 탐색 (OI-U2)**:

1. 3가지 노이즈 조건에서 실시간 노이즈/신호 비율 로그를 수집합니다 (10분 × 3조건).
2. 조건별 비율 분포 히스토그램 그리기; 낮은 노이즈 정상 범위의 99번째 백분위수와 높은 노이즈 범위의 1번째 백분위수 간의 간격 확인.
3. 3–5가지 후보 임계값을 적용하여 조건별 오탐률과 누락률을 측정합니다.
4. 오탐률 + 누락률의 가중 합계가 최소인 최종 임계값 선택.
5. QAS-4 응답 측도에 N·M 값 + 노이즈/신호 임계값 기록.
