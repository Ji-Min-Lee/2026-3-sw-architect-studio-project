# Architectural Drivers — TimeGrapher

**Milestone**: M1 | **Due**: 2026-06-09 | **Updated**: 2026-06-02

## 1. Project Context & Objectives

### 한국어

TimeGrapher는 기계식 시계의 탈진기 진동에서 발생하는 음향 신호(beat noise)를 분석하여 시계 성능을 실시간으로 진단하는 시스템이다. 마이크로 시계의 tick-tock 소리를 수집하고, A(T1)·C(T3) 이벤트를 감지하여 Rate, Amplitude, Beat Error를 계산하고 GUI에 표시한다.

이 프로젝트의 핵심 목표는 다음과 같다:

| 목표 | 설명 |
|------|------|
| 실시간 진단 | Raspberry Pi 5 위에서 음향 신호를 실시간으로 수집·처리·표시 |
| 정확한 측정 | Rate, Amplitude, Beat Error 값이 WeiShi No.1000 레퍼런스 디바이스와 일치 |
| 확장 가능한 구조 | 새로운 그래프·분석 기능을 기존 코드 수정 없이 추가 가능 |
| 아키텍처 원칙 실증 | CMU MSE 소프트웨어 아키텍처 설계 원칙을 적용한 시스템 설계 |

아키텍처 드라이버는 이 목표에서 직접 도출된다. "실시간 진단"은 Real-Time Performance와 Low Latency로, "정확한 측정"은 Measurement Accuracy와 Correctness로, "확장 가능한 구조"는 Extensibility로 구체화된다.

### English

TimeGrapher is a system that analyzes the acoustic signal (beat noise) generated by the escapement vibration of a mechanical watch to diagnose watch performance in real time. It captures tick-tock sounds via a microphone, detects A(T1)·C(T3) events, calculates Rate, Amplitude, and Beat Error, and displays them on a GUI.

The core objectives of this project are:

| Objective | Description |
|-----------|-------------|
| Real-time diagnostics | Capture, process, and display acoustic signals in real time on Raspberry Pi 5 |
| Accurate measurement | Rate, Amplitude, and Beat Error values match the WeiShi No.1000 reference device |
| Extensible structure | New graphs and analysis features can be added without modifying existing code |
| Architecture demonstration | Apply CMU MSE software architecture design principles to a real system |

The architectural drivers are derived directly from these objectives. "Real-time diagnostics" maps to Real-Time Performance and Low Latency; "accurate measurement" maps to Measurement Accuracy and Correctness; "extensible structure" maps to Extensibility.

## 3. Quality Attribute Requirements — Priority & Rationale

### 한국어

5개의 QA를 아래 순서로 우선순위를 결정했다. 우선순위는 "아키텍처 결정에 가장 큰 제약을 주는 순서"를 기준으로 한다.

| 순위 | QA | 한 줄 근거 |
|------|-----|---------|
| **1** | Real-Time Performance | 나머지 모든 QA의 선행 조건. 이것이 무너지면 데이터 자체가 없음 |
| **2** | Measurement Accuracy | 프로젝트의 존재 이유. 아키텍처 결정(sps, 감지 알고리즘)에 가장 큰 영향 |
| **3** | Low Latency | Beat 주기라는 hard threshold 존재. 초과 시 기능 붕괴. 실험으로만 검증 가능 |
| **4** | Correctness | Accuracy + 단일 데이터 소스 구조로 자동 확보. 독립적 아키텍처 목표가 아님 |
| **5** | Extensibility | 개발자 편의성이지만 11개 그래프 구현 일정 리스크를 직접 통제 |

```
Real-Time Performance   → 시스템이 살아있는가?           (존재 조건)
Measurement Accuracy    → 시스템이 올바른가?              (목적 달성)
Low Latency             → 시스템이 쓸 만한가?             (사용자 체감)
Correctness             → 시스템이 내부적으로 일관적인가?  (신뢰성)
Extensibility           → 시스템을 계속 만들 수 있는가?   (개발 지속성)
```

**우선순위 판단의 핵심 원칙**: Real-Time Performance가 무너지면 dropped audio block이 발생하고 A·C 이벤트 자체가 누락된다. 이벤트가 없으면 나머지 4개 QA를 측정할 데이터 자체가 존재하지 않는다. Measurement Accuracy가 Extensibility보다 높은 이유는, 아키텍처가 주로 "정확한 감지"를 중심으로 설계되어야 하기 때문이다. Correctness가 4순위인 이유는 독립적으로 달성해야 할 목표가 아니라 Accuracy와 좋은 설계가 결합될 때 자연스럽게 얻어지는 속성이기 때문이다.

### English

Five QAs are prioritized in the order below, using "degree of constraint on architectural decisions" as the criterion.

| Priority | QA | One-line rationale |
|----------|-----|-------------------|
| **1** | Real-Time Performance | Precondition for all other QAs. Without it, no data exists |
| **2** | Measurement Accuracy | The reason the project exists. Greatest influence on architectural decisions (sps, detection algorithm) |
| **3** | Low Latency | Hard threshold defined by beat interval. Exceeding it causes functional failure. Only verifiable through experiments |
| **4** | Correctness | Automatically secured by Accuracy + single data source structure. Not an independent architectural target |
| **5** | Extensibility | Developer-facing, but directly controls schedule risk across 11 graph implementations |

```
Real-Time Performance   → Is the system alive?               (existence condition)
Measurement Accuracy    → Is the system correct?             (purpose achieved)
Low Latency             → Is the system usable?              (user experience)
Correctness             → Is the system internally coherent? (trustworthiness)
Extensibility           → Can development continue?          (development sustainability)
```

**Key prioritization principle**: If Real-Time Performance fails, dropped audio blocks occur and A·C events are missed. Without events, there is no data to measure any of the other four QAs. Measurement Accuracy ranks above Extensibility because the architecture must be designed primarily around accurate detection. Correctness ranks 4th because it is not an independent goal — it is a property that emerges naturally when Accuracy is secured and the architecture uses a single data source.

## 4. Quality Attribute Scenarios

### QA-1: Real-Time Performance ★ Priority 1

**한국어**

| 항목 | 내용 |
|------|------|
| **Source** | 기계식 시계 (음향 신호 발생) |
| **Stimulus** | USB 센서를 통해 연속적인 음향 샘플 스트림 입력 |
| **Artifact** | 신호 수집 → 처리 → 분석 → GUI 표시 파이프라인 전체 |
| **Environment** | Raspberry Pi 5 (8GB RAM), 정상 운영 상태 |
| **Response** | 수집된 샘플을 끊김 없이 실시간으로 처리하고 GUI에 표시 |
| **Response Measure** | - **Objective**: 96,000 sps 처리 유지<br>- **Minimum**: 48,000 sps (이 이하면 프로젝트 실패)<br>- **Stretch**: 192,000 sps<br>- Dropped audio block: **0개** |

**English**

| Field | Content |
|-------|---------|
| **Source** | Mechanical watch (acoustic signal source) |
| **Stimulus** | Continuous stream of acoustic samples from USB sensor |
| **Artifact** | Entire signal acquisition → processing → analysis → GUI display pipeline |
| **Environment** | Raspberry Pi 5 (8GB RAM), normal operating conditions |
| **Response** | Captured samples processed in real time without interruption and displayed on GUI |
| **Response Measure** | - **Objective**: sustain 96,000 sps processing<br>- **Minimum**: 48,000 sps (below this = project failure)<br>- **Stretch**: 192,000 sps<br>- Dropped audio blocks: **0** |

**수치 도출 근거 / Derivation**

```
28,800 BPH 시계 기준: 초당 8 beat, 96k sps → beat당 12,000샘플
43,200 BPH 시계 기준: 초당 12 beat, 96k sps → beat당 8,000샘플

sps별 T1/T3 감지 정밀도:
  48,000 sps → 1샘플 = 0.021ms 오차
  96,000 sps → 1샘플 = 0.010ms 오차  ← 목표
  192,000 sps → 1샘플 = 0.005ms 오차

Dropped block 발생 시: A·C 이벤트 누락 → Rate·Amplitude·Beat Error 계산 불가
```

**아키텍처 영향 / Architectural Implication**

이 QA가 가장 먼저 해결해야 할 설계 문제를 정의한다. RPi 5에서 96k sps + 11개 그래프 렌더링을 동시에 처리하려면 파이프라인 구조와 스레드 분리가 필수적이다.

This QA defines the first design problem to solve. Sustaining 96k sps while rendering 11 graphs simultaneously on RPi 5 mandates a pipeline structure with thread separation.

### QA-2: Measurement Accuracy ★ Priority 2

**한국어**

| 항목 | 내용 |
|------|------|
| **Source** | 기계식 시계 (escapement 진동 발생) |
| **Stimulus** | T1(impulse), T3(lock+banking) 이벤트 포함 음향 신호 입력 |
| **Artifact** | 신호 처리 파이프라인의 beat event detection 모듈 |
| **Environment** | 정상 운영 상태 / 신호가 약하거나 노이즈가 있는 열화 환경 |
| **Response** | T1·T3 onset/peak를 정확히 감지하여 Rate, Amplitude, Beat Error 계산. 신호 열화 시 불안정한 값 대신 명확한 경고 표시 |
| **Response Measure** | - WeiShi No.1000 대비 Rate 오차: **< 5 s/d** *(잠정, Experiment 3 후 확정)*<br>- WeiShi No.1000 대비 Beat Error 오차: **< 0.1 ms** *(잠정, Experiment 3 후 확정)*<br>- 신호 열화 시 graceful degradation (불안정 출력 없음) |

**English**

| Field | Content |
|-------|---------|
| **Source** | Mechanical watch (escapement vibration source) |
| **Stimulus** | Acoustic signal containing T1 (impulse) and T3 (lock+banking) events |
| **Artifact** | Beat event detection module in the signal processing pipeline |
| **Environment** | Normal operating conditions / degraded environment (weak signal or noise) |
| **Response** | Accurately detect T1·T3 onset/peak and compute Rate, Amplitude, Beat Error. On signal degradation, display a clear warning instead of unstable values |
| **Response Measure** | - Rate error vs. WeiShi No.1000: **< 5 s/d** *(provisional, confirmed after Experiment 3)*<br>- Beat Error error vs. WeiShi No.1000: **< 0.1 ms** *(provisional, confirmed after Experiment 3)*<br>- Graceful degradation on signal degradation (no unstable output) |

**수치 도출 근거 / Derivation**

```
Rate < 5 s/d:
  Witschi 도메인 기준 — 기계식 시계 정상 운영 범위 ±5 s/d
  시스템 오차 = 5 s/d이면 정상 시계를 비정상으로 오진단 가능
  → 출발점이자 상한. Experiment 3 후 더 엄격하게 조정 예정

Beat Error < 0.1 ms:
  도메인 정상 범위 ≤ 0.6 ms의 약 1/6 수준
  시스템 오차는 측정 대상 허용 범위보다 충분히 작아야 신뢰 가능
```

**아키텍처 영향 / Architectural Implication**

sps 선택, 필터 설계, T1/T3 감지 알고리즘이 모두 이 QA를 중심으로 결정된다. C(T3) 이벤트는 A보다 작고 불규칙하며 Amplitude 계산 분모(t_AC)에 들어가므로 오차 증폭이 발생한다. 감지 알고리즘의 품질이 핵심 아키텍처 결정이다.

sps selection, filter design, and T1/T3 detection algorithm are all determined around this QA. C(T3) events are smaller and more irregular than A, and appear in the Amplitude formula denominator (t_AC), causing error amplification. Detection algorithm quality is the central architectural decision.

### QA-3: Low Latency ★ Priority 3

**한국어**

| 항목 | 내용 |
|------|------|
| **Source** | 기계식 시계 (beat 이벤트 발생) |
| **Stimulus** | 마이크에서 음향 샘플 블록 캡처 시작 |
| **Artifact** | capture → beat detection → 계산 → GUI 렌더링 전 구간 |
| **Environment** | Raspberry Pi 5, 실시간 운영 상태 |
| **Response** | 캡처된 샘플에 대한 파형, 마커, 계산값이 GUI에 표시됨 |
| **Response Measure** | - ① capture→process: **< 46 ms** *(잠정)*<br>- ② process→display: **< 20 ms** *(잠정)*<br>- ③ end-to-end: **< 66 ms** *(잠정, 43,200 BPH 기준)*<br>- Dropped audio block: 0, Missed beat: 0<br>*(Experiment 1·2 + BPH 스코프 팀 합의 후 확정)* |

**English**

| Field | Content |
|-------|---------|
| **Source** | Mechanical watch (beat event source) |
| **Stimulus** | Acoustic sample block captured from microphone |
| **Artifact** | Entire pipeline: capture → beat detection → calculation → GUI rendering |
| **Environment** | Raspberry Pi 5, real-time operating conditions |
| **Response** | Waveform, markers, and computed values for the captured samples displayed on GUI |
| **Response Measure** | - ① capture→process: **< 46 ms** *(provisional)*<br>- ② process→display: **< 20 ms** *(provisional)*<br>- ③ end-to-end: **< 66 ms** *(provisional, based on 43,200 BPH)*<br>- Dropped audio blocks: 0, Missed beats: 0<br>*(To be confirmed after Experiment 1·2 + BPH scope team consensus)* |

**수치 도출 근거 / Derivation**

```
지원 BPH 범위에 따른 beat 주기:
  28,800 BPH → 125ms   36,000 BPH → 100ms   43,200 BPH → 83ms

현재 기준: 43,200 BPH (worst-case)
  end-to-end 목표: 83ms × 80% = 66ms  (안전 마진: OS 지터, Qt 렌더링 변동 흡수)
  ① capture→process: 66ms의 ~70% → 46ms
  ② process→display: 66ms의 ~30% → 20ms

⚠️ 팀 결정 필요: 지원 BPH 범위 확정 (보유 시계 BPH 확인 필요)
  옵션 A — 28,800 BPH까지: end-to-end < 100ms
  옵션 B — 43,200 BPH까지: end-to-end < 66ms  ← 현재 잠정
```

**Latency가 단순 체감이 아닌 이유 / Why latency is not just comfort**

end-to-end가 beat 주기를 초과하면 다음 beat가 이미 도착한 상태에서 이전 beat를 처리하게 된다. Backlog가 누적되면 dropped beat로 이어지고 이는 사실상 Real-Time Performance 실패와 동일한 결과를 낳는다.

If end-to-end latency exceeds the beat interval, the system processes a beat while the next has already arrived. Accumulated backlog leads to dropped beats — effectively the same outcome as a Real-Time Performance failure.

### QA-4: Correctness ★ Priority 4

**한국어**

| 항목 | 내용 |
|------|------|
| **Source** | 사용자 (동일 시계, 동일 조건에서 측정) |
| **Stimulus** | 실시간 음향 신호 수신 중 다수의 GUI 뷰 동시 표시 |
| **Artifact** | Rate, Amplitude, Beat Error 계산 로직 및 GUI 전체 뷰 |
| **Environment** | 정상 운영 상태 / ambient noise가 존재하는 환경 |
| **Response** | 모든 GUI 뷰에서 동일 데이터 기반으로 일관된 수치 표시 |
| **Response Measure** | - 동일 beat 데이터에서 파생된 모든 뷰의 수치 편차: **0** (허용 없음)<br>- 단일 데이터 소스 구조로 구조적 보장 |

**English**

| Field | Content |
|-------|---------|
| **Source** | User (measuring with the same watch under the same conditions) |
| **Stimulus** | Multiple GUI views displayed simultaneously while receiving real-time acoustic signal |
| **Artifact** | Rate, Amplitude, Beat Error calculation logic and all GUI views |
| **Environment** | Normal operating conditions / environment with ambient noise |
| **Response** | All GUI views display consistent values derived from the same data |
| **Response Measure** | - Value deviation across all views derived from the same beat data: **0** (none permitted)<br>- Structurally guaranteed by single data source architecture |

**Correctness vs. Measurement Accuracy 구분 / Distinction**

```
Correctness:        뷰들이 서로 일치하는가? (내부 일관성)
Measurement Accuracy: 뷰의 값이 WeiShi와 일치하는가? (외부 정확성)

Correctness ✓ + Accuracy ✗ 가능:
  모든 뷰가 일관되게 +10 s/d를 표시하지만 WeiShi는 +3 s/d인 경우
  → Correctness는 만족하지만 시스템은 쓸모없음

따라서 Accuracy가 더 높은 우선순위를 가짐.
Correctness는 단일 데이터 소스 구조를 채택하면 구조적으로 보장됨
→ 별도 아키텍처 목표가 아니라 설계 원칙의 결과물
```

```
Correctness:          do the views agree with each other? (internal consistency)
Measurement Accuracy: do the values agree with WeiShi?   (external correctness)

Correctness ✓ + Accuracy ✗ is possible:
  All views consistently show +10 s/d, but WeiShi shows +3 s/d
  → Correctness is satisfied but the system is useless

Therefore Accuracy holds higher priority.
Correctness is structurally guaranteed by adopting a single data source
→ not an independent architectural target, but a consequence of good design
```

### QA-5: Extensibility ★ Priority 5

**한국어**

| 항목 | 내용 |
|------|------|
| **Source** | 개발자 (새 그래프 또는 분석 기능 추가) |
| **Stimulus** | 기존 코드베이스에 새로운 그래프/필터/디스플레이 모드 추가 요청 |
| **Artifact** | GUI 모듈 구조 전체 (신호 수집 / 처리 / 계산 / 표시 레이어) |
| **Environment** | 개발 환경 (PC 또는 RPi), 개발 기간 내 |
| **Response** | 기존 모듈의 major redesign 없이 새 기능을 독립적으로 추가·테스트 가능 |
| **Response Measure** | - 새 그래프 1개 추가 시 변경 파일 수: **≤ 3개** *(잠정, 코드 분석 후 팀 합의)*<br>- 기존 계산 로직 모듈 수정 없이 표시 레이어만 추가 가능 |

**English**

| Field | Content |
|-------|---------|
| **Source** | Developer (adding new graph or analysis feature) |
| **Stimulus** | Request to add new graph / filter / display mode to the existing codebase |
| **Artifact** | Entire GUI module structure (acquisition / processing / calculation / display layers) |
| **Environment** | Development environment (PC or RPi), during the development period |
| **Response** | New features can be added and tested independently without major redesign of existing modules |
| **Response Measure** | - Files changed when adding 1 new graph: **≤ 3** *(provisional, confirmed after codebase analysis + team consensus)*<br>- Addition possible by touching only the display layer, without modifying calculation logic |

**수치 도출 근거 / Derivation**

```
≤ 3개 파일의 근거:
  이상적인 플러그인 구조에서 불가피한 최솟값:
    ① 새 그래프 위젯 파일 (신규 생성)       → 1개
    ② 탭 패널에 새 탭 등록 (기존 수정)      → 1개
    ③ 데이터 구독 연결 (기존 수정)          → 1개
                                            합계: 3개

  이보다 많으면 → 구조가 충분히 모듈화되지 않았다는 신호
  11개 그래프를 Week 3~4에 구현해야 하므로 이 제약이 일정 리스크를 통제함
  최종 데모에서 "3개 파일만 변경"은 Extensibility를 증명하는 구체적 수치
```

```
Rationale for ≤ 3 files:
  Unavoidable minimum in an ideal plugin structure:
    ① New graph widget file (created new)          → 1 file
    ② Tab panel registration (existing modified)   → 1 file
    ③ Data subscription wiring (existing modified) → 1 file
                                              Total: 3 files

  More than 3 → signals insufficient modularity
  11 graphs must be implemented in Weeks 3–4; this constraint controls schedule risk
  "Only 3 files changed" is a concrete, demonstrable claim at the final demo
```

## 7. Design Constraints

### 한국어

> Design Constraint = 이미 내려진 설계 결정. 아키텍처 논의 대상이 아니며 변경 불가.

| 제약 | 내용 | 변경 불가 이유 |
|------|------|-------------|
| 플랫폼 | Raspberry Pi 5 (ARM64, 8GB RAM) | 프로젝트에서 하드웨어 지급 |
| 언어/프레임워크 | C++ / Qt (Qt Creator IDE) | 기존 코드베이스 언어 |
| 기존 코드베이스 | TimeGrapher_v10.5를 교체가 아닌 확장 | 프로젝트 요구사항 명시 |
| AGC | AlsaMixer에서 Auto Gain Control 비활성화 | 켜진 상태에서는 신호 왜곡 → Accuracy 달성 불가 |

### English

> Design Constraint = a design decision already made. Not subject to architectural discussion; cannot be changed.

| Constraint | Description | Why non-negotiable |
|------------|-------------|-------------------|
| Platform | Raspberry Pi 5 (ARM64, 8GB RAM) | Hardware provided by the project |
| Language / Framework | C++ / Qt (Qt Creator IDE) | Language of the existing codebase |
| Baseline code | Must extend TimeGrapher_v10.5, not replace it | Explicitly required by project spec |
| AGC | Auto Gain Control must be disabled in AlsaMixer | Signal distortion when enabled → Accuracy target unreachable |
