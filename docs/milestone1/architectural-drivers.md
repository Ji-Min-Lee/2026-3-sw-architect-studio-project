## 1. 프로젝트 맥락 및 목표 / Project Context & Objectives

**한국어**

TimeGrapher는 기계식 시계의 탈진기 진동에서 발생하는 음향 신호(beat noise)를 분석하여 시계 성능을 실시간으로 진단하는 시스템이다. 마이크로 시계의 tick-tock 소리를 수집하고, A(T1)·C(T3) 이벤트를 감지하여 Rate, Amplitude, Beat Error를 계산하고 GUI에 표시한다.

| 목표 / Goal | 설명 / Description |
|------------|------------------|
| 실시간 진단 / Real-Time Diagnosis | Raspberry Pi 5 위에서 음향 신호를 실시간으로 수집·처리·표시 / Collect, process, and display acoustic signals in real time on Raspberry Pi 5 |
| 정확한 측정 / Accurate Measurement | Rate, Amplitude, Beat Error 값이 WeiShi No.1000 레퍼런스 디바이스와 일치 / Rate, Amplitude, Beat Error values match the WeiShi No.1000 reference device |
| 확장 가능한 구조 / Extensible Architecture | 새로운 그래프·분석 기능을 기존 코드 수정 없이 추가 가능 / New graphs and analysis features can be added without modifying existing code |
| 아키텍처 원칙 실증 / Architecture Principles | CMU MSE 소프트웨어 아키텍처 설계 원칙을 적용한 시스템 설계 / System design applying CMU MSE software architecture principles |

**English**

TimeGrapher is a system that analyzes acoustic signals (beat noise) generated by the escapement vibration of a mechanical watch to diagnose watch performance in real time. It captures the tick-tock sound from a microphone, detects A(T1)·C(T3) events, computes Rate, Amplitude, and Beat Error, and displays them in a GUI.

## 3. 품질 속성 시나리오 / Quality Attribute Scenarios

### 3.1 우선순위 및 근거 / Priority & Rationale

**한국어**

5개의 QA를 "아키텍처 결정에 가장 큰 제약을 주는 순서"로 우선순위를 결정했다.

| 순위 / Rank | QA | 한 줄 근거 / One-Line Rationale |
|:-----------:|----|-------------------------------|
| **1** | Real-Time Performance | 나머지 모든 QA의 선행 조건. 이것이 무너지면 데이터 자체가 없음 |
| **2** | Measurement Accuracy | 프로젝트의 존재 이유. 아키텍처 결정(sps, 감지 알고리즘)에 가장 큰 영향 |
| **3** | Low Latency | Beat 주기라는 hard threshold 존재. 초과 시 기능 붕괴. 실험으로만 검증 가능 |
| **4** | Correctness | Accuracy + 단일 데이터 소스 구조로 자동 확보. 독립적 아키텍처 목표가 아님 |
| **5** | Extensibility | 개발자 편의성이지만 11개 그래프 구현 일정 리스크를 직접 통제 |

**English**

The five QAs are prioritized in order of the greatest constraint they impose on architectural decisions.

| Rank | QA | One-Line Rationale |
|:----:|----|-------------------|
| **1** | Real-Time Performance | Prerequisite for all other QAs — without it, there is no data |
| **2** | Measurement Accuracy | The reason this project exists; most directly shapes architectural decisions (sps, detection algorithm) |
| **3** | Low Latency | Hard threshold defined by beat period; exceeding it causes functional failure; verifiable only through experiment |
| **4** | Correctness | Structurally guaranteed by Accuracy + single data source; not an independent architectural goal |
| **5** | Extensibility | Developer convenience, but directly controls schedule risk for implementing 11 graphs |

#### 수치 근거 / Latency Target Derivation

**한국어**

**핵심 제약**: end-to-end 지연이 beat 주기를 초과하면 다음 beat 처리 전에 이전 beat 표시가 완료되지 않아 기능 붕괴.  
**안전 마진 80%** 적용 (OS 지터, Qt 렌더링 변동 흡수 목적).

| BPH | Beat 주기 / Period | 80% 마진 → end-to-end 목표 | ① capture→process (70%) | ② process→display (30%) |
|-----|------------------|--------------------------|------------------------|------------------------|
| 28,800 | 125 ms | 100 ms | 70 ms | 30 ms |
| 36,000 | 100 ms | 80 ms | 56 ms | 24 ms |
| **43,200** | **83 ms** | **66 ms** | **46 ms** | **20 ms** |

> ⚠️ **팀 결정 필요 — 지원 BPH 범위 미확정 (보유 시계 BPH 확인 후 결정)**
>
> | 옵션 / Option | 지원 범위 / Range | end-to-end | ① | ② |
> |:------------:|-----------------|-----------|---|---|
> | **Option A** | ~28,800 BPH | < 100 ms | < 70 ms | < 30 ms |
> | **Option B** *(현재 잠정 / current provisional)* | ~43,200 BPH (worst-case) | **< 66 ms** | **< 46 ms** | **< 20 ms** |
>
> *현재 잠정값은 Option B (worst-case 43,200 BPH 기준). EX-01 결과 및 보유 시계 BPH 확인 후 팀 합의로 확정.*

**English**

**Core constraint**: If end-to-end latency exceeds the beat period, the previous beat display cannot complete before the next beat is processed, causing functional failure.  
**80% safety margin** applied to absorb OS jitter and Qt rendering variation.

> ⚠️ **Team decision required — supported BPH range not yet confirmed (to be decided after verifying available watches)**
>
> *Current provisional values use Option B (worst-case 43,200 BPH). To be finalized by team after EX-01 results and available watch BPH confirmation.*

**공통 제약 / Common Constraints**: Dropped audio block: 0, Missed beat: 0  
*([EX-01](./planned-experiments.md#ex-01-rpi-성능-벤치마크--rpi-performance-benchmark) 결과 후 확정 / to be confirmed after EX-01)*

#### 수치 근거 / "≤ 3 Files" Derivation

**한국어**

**≤ 3개는 이상적인 플러그인 구조에서 이론적 최솟값이다.**

| 변경 유형 / Change Type | 대상 / Target | 파일 수 / Files |
|------------------------|--------------|:--------------:|
| ① 새 그래프 위젯 파일 신규 생성 / New graph widget file | `GraphFoo.cpp` / `GraphFoo.h` | 1 |
| ② 탭 패널에 새 탭 등록 / Register tab in panel | `TabPanel.cpp` (기존 수정 / existing) | 1 |
| ③ 데이터 구독 연결 / Wire data subscription | `DataBroker.cpp` (기존 수정 / existing) | 1 |
| **합계 / Total** | | **3** |

- **하한(floor)**: 0개 파일로 그래프 추가 불가 — 최소 1개 신규 생성 필수
- **초과 시 의미**: 4개 이상이면 모듈화가 불충분하다는 구조적 신호 (ex. God Object 수정 강제)
- **일정 연결**: 11개 그래프를 Week 3~4에 구현해야 하므로 이 제약이 Extensibility를 수치로 강제하고 일정 리스크를 직접 통제함
- **데모 검증**: "그래프 1개 추가 = 파일 3개 변경"은 최종 데모에서 Extensibility를 증명하는 구체적 측정 기준

**English**

**≤ 3 is the theoretical minimum in an ideal plugin-style architecture.**

- **Floor**: Cannot add a graph with 0 file changes — at least 1 new file must be created
- **If exceeded**: ≥ 4 files signals insufficient modularization (e.g., God Object forces modification)
- **Schedule link**: With 11 graphs due in Weeks 3–4, this constraint numerically enforces Extensibility and directly controls schedule risk
- **Demo verification**: "1 graph added = 3 files changed" is a concrete, measurable criterion for demonstrating Extensibility at the final demo

## 4. 설계 제약사항 / Design Constraints

**한국어**

Design Constraint = 이미 내려진 설계 결정. 아키텍처 논의 대상이 아니며 변경 불가.  
언어·프레임워크·라이브러리는 팀 선택 사항이므로 제약에 포함하지 않는다.

**English**

Design Constraints are decisions already made. They are not subject to architectural discussion and cannot be changed.  
Language, framework, and library choices are team decisions and are therefore not listed as constraints.

| 제약 / Constraint | 내용 / Detail | 변경 불가 이유 / Rationale |
|------------------|-------------|--------------------------|
| 타겟 하드웨어 / Target Hardware | Raspberry Pi 5 (ARM64, 8GB RAM) + USB audio sensor + 8" touchscreen | 프로젝트에서 하드웨어 지급 / Hardware provided by program |
| 레퍼런스 디바이스 / Reference Device | WeiShi No.1000 | 평가 기준 장비 — 정확도 측정의 ground truth / Grading reference device — ground truth for accuracy |
