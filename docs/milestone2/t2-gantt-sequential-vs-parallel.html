T2 DSP Offload Thread — Sequential vs Parallel Execution

EXP-02 · macOS 96 kHz · R1 Baseline (before) vs R2 T2 (after) · feature/layer-ex-baseline

Before T2 — Sequential (Main-Thread DSP) · R1 Baseline
AudioWorker (BG) emit
Qt queue wait
DSP exec
Tab render (sync · RPi)
BG emits a signal every ~10 ms, but the Main Thread is busy with UI events and can only dequeue it after a long wait (avg 420 ms). DSP and rendering then execute sequentially on the same thread, blocking the event loop. Subsequent BG signals pile up in the queue → backlog 47 %.
After T2 — Parallel (DSP Offload Thread) · R2
AudioWorker (BG) emit
DSP exec (DSP thread)
FG queue wait (fg_wait)
FG handle (Main thread)
DSPWorker runs in a dedicated thread with its own event loop. It picks up each BG signal immediately (avg 0.013 ms wait). DSP and FG processing run in parallel — DSP finishes within 0.4 ms, well before the next BG signal arrives at 10 ms. Backlog drops to 0 %. The only new cost is fg_wait (avg 8.9 ms): the Qt scheduler wakes the Main Thread after DSPWorker emits frameLogged.
